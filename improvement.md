# Описание организации базы данных и ТЗ на улучшение

## 1. Текущее состояние

Проект использует MySQL 8 как основное хранилище данных для публичного сайта отеля. База данных по умолчанию называется `resort_clone`. Подключение настраивается через переменные окружения `MYSQL_DSN` или через отдельные параметры `MYSQL_USER`, `MYSQL_PASSWORD`, `MYSQL_HOST`, `MYSQL_PORT`, `MYSQL_DATABASE`.

В Docker Compose база поднимается как сервис `db` на образе `mysql:8.4`, хранит данные в named volume `mysql_data` и проверяется healthcheck-командой `mysqladmin ping`. Приложение подключается к базе через сервисное имя `db`.

Схема описана в двух местах:

- `db/schema.sql` - SQL-файл с базовой структурой таблиц.
- `internal/repository/mysql.go` - метод `EnsureSchema`, который на старте приложения создает таблицы и добавляет некоторые недостающие колонки.

При запуске приложение выполняет такую последовательность:

1. Загружает конфигурацию.
2. Открывает пул MySQL-соединений.
3. Вызывает `EnsureSchema`.
4. Вызывает `SeedDefaults`.
5. Запускает HTTP-сервер.

Пул соединений настроен в коде:

- `ConnMaxLifetime`: 5 минут.
- `MaxOpenConns`: 10.
- `MaxIdleConns`: 10.

## 2. Таблицы

### 2.1. `site_content`

Назначение: хранит глобальный контент главной страницы и контактные данные.

Текущая модель является singleton-style: приложение читает первую строку по `ORDER BY id ASC LIMIT 1` и считает ее активной. При этом на уровне базы данных нет ограничения, которое запрещает создать несколько строк.

Основные поля:

- `id` - первичный ключ.
- `brand_name` - название отеля/бренда.
- `tagline` - короткий слоган.
- `hero_title` - основной заголовок первого экрана.
- `hero_subtitle` - подзаголовок первого экрана.
- `hero_image_path` - путь к изображению hero-блока.
- `season_note` - сезонная заметка.
- `booking_url` - ссылка для бронирования.
- `address`, `phone`, `email` - контактные данные.
- `check_in`, `check_out` - время заезда и выезда.
- `about_title`, `about_body` - блок описания отеля.
- `created_at`, `updated_at` - технические timestamps.

Особенности:

- Используется как единая таблица для разных смысловых блоков: бренд, hero, контакты, about, параметры заезда.
- Контент обновляется методом `UpdateSiteContent`.
- Данные сидируются только если таблица пустая.
- Время заезда/выезда хранится как `VARCHAR(32)`, а не как тип `TIME`.
- URL, email, телефон и image path не валидируются на уровне БД.

### 2.2. `rooms`

Назначение: хранит карточки категорий номеров, которые выводятся на главной странице.

Основные поля:

- `id` - первичный ключ.
- `name` - название категории номера.
- `summary` - описание.
- `capacity` - вместимость.
- `size_sqm` - площадь в квадратных метрах.
- `room_count` - количество комнат/единиц в категории.
- `feature_note` - короткая заметка/бейдж.
- `image_path` - путь к изображению номера.

Особенности:

- Все строки читаются по `ORDER BY id ASC`.
- Нет отдельного поля порядка сортировки.
- Нет `created_at` и `updated_at`.
- Нет ограничений на положительные значения `capacity`, `size_sqm`, `room_count`.
- Нет признаков публикации/скрытия.

### 2.3. `gallery_items`

Назначение: хранит элементы галереи главной страницы.

Основные поля:

- `id` - первичный ключ.
- `title` - заголовок карточки.
- `subtitle` - описание.
- `tone` - ключ fallback-оформления, если нет изображения.
- `image_path` - путь к изображению.

Особенности:

- Все строки читаются по `ORDER BY id ASC`.
- Нет отдельного поля порядка сортировки.
- Нет `created_at` и `updated_at`.
- Значение `tone` не ограничено справочником или `CHECK`.
- Нет признаков публикации/скрытия.

### 2.4. `scrape_snapshots`

Назначение: хранит снимки исходных страниц для справки и аудита.

Основные поля:

- `id` - первичный ключ.
- `source_url` - URL источника.
- `page_title` - заголовок исходной страницы.
- `description` - описание.
- `markdown` - нормализованное markdown-представление.
- `html` - сохраненный HTML.
- `created_at` - время создания снимка.

Особенности:

- Таблица используется как append-only история.
- Главная страница читает последний снимок через `ORDER BY id DESC LIMIT 1`.
- Нет индекса по `created_at`.
- Нет статуса обработки, хеша содержимого, размера документа или признака актуальности.
- `html` и `markdown` хранятся прямо в MySQL как `MEDIUMTEXT`, что может быстро увеличить размер базы.

## 3. Текущая модель чтения и записи

Главная страница загружает агрегат `HomePageData`:

- первую строку `site_content`;
- все строки `rooms`;
- все строки `gallery_items`;
- последний `scrape_snapshots`, если он есть.

Записи поддерживаются на уровне репозитория:

- обновление `site_content`;
- создание, обновление и удаление `rooms`;
- создание, обновление и удаление `gallery_items`;
- сохранение `scrape_snapshots`.

Админ-интерфейс в текущем проекте отсутствует, но README указывает, что модель данных оставлена с запасом под отдельное админ-приложение.

## 4. Основные проблемы текущей организации

1. Схема дублируется в `db/schema.sql` и `internal/repository/mysql.go`.
2. Нет полноценной системы версионированных миграций.
3. Singleton-подход для `site_content` не закреплен ограничениями БД.
4. Порядок отображения комнат и галереи зависит от `id`.
5. У большинства контентных таблиц нет `created_at` и `updated_at`.
6. Нет `is_active` или `published_at` для управления публикацией.
7. Нет ограничений целостности для числовых полей.
8. Нет индексов под реальные запросы, кроме первичных ключей.
9. Пути к изображениям хранятся строками в разных таблицах, без единой модели медиа.
10. `scrape_snapshots` хранит большие документы в основной базе без политики очистки.
11. Нет внешних ключей, потому что текущие таблицы почти не нормализованы.
12. Нет отдельной модели отеля/объекта размещения, из-за чего база плохо масштабируется на несколько объектов.
13. Контент частично хранится в БД, частично зашит в шаблонах.
14. Нет истории изменений контента и аудита действий администратора.
15. Нет тестов, которые проверяют актуальность SQL-схемы и миграций.

## 5. ТЗ на улучшение организации базы данных

### 5.1. Цель

Сделать базу данных проекта надежной, расширяемой и удобной для будущего администрирования сайта отеля: убрать дублирование схемы, ввести миграции, закрепить бизнес-ограничения на уровне БД, улучшить сортировку, публикацию, аудит и хранение медиа.

### 5.2. Этап 1. Ввести версионированные миграции

Задачи:

1. Выбрать инструмент миграций для Go-проекта, например `golang-migrate/migrate` или `pressly/goose`.
2. Создать каталог `db/migrations`.
3. Перенести текущую схему из `db/schema.sql` в первую миграцию.
4. Убрать дублирование `CREATE TABLE` из `EnsureSchema` или заменить `EnsureSchema` запуском миграций.
5. Оставить `db/schema.sql` только как генерируемый snapshot схемы или удалить его, если он больше не нужен.
6. Добавить инструкцию запуска миграций в README.
7. Добавить миграции в Docker/startup workflow.

Критерии приемки:

- База с нуля создается через миграции.
- Повторный запуск миграций не ломает приложение.
- Источник правды по структуре БД только один.
- В README описано, как применить и откатить миграции.

Приоритет: высокий.

### 5.3. Этап 2. Закрепить singleton-модель `site_content`

Задачи:

1. Добавить поле `content_key VARCHAR(64) NOT NULL DEFAULT 'main'`.
2. Добавить уникальный индекс `uniq_site_content_key(content_key)`.
3. Изменить чтение активной записи с `ORDER BY id ASC LIMIT 1` на выборку по `content_key = 'main'`.
4. Обновить seed-логику, чтобы она создавала запись с `content_key = 'main'`.
5. Добавить миграцию, которая аккуратно проставит `content_key` существующей первой записи.

Критерии приемки:

- Нельзя создать две активные строки с `content_key = 'main'`.
- Приложение больше не зависит от минимального `id`.
- Старые данные сохраняются после миграции.

Приоритет: высокий.

### 5.4. Этап 3. Добавить сортировку и публикацию контента

Задачи:

1. В таблицы `rooms` и `gallery_items` добавить:
   - `sort_order INT NOT NULL DEFAULT 100`;
   - `is_active BOOLEAN NOT NULL DEFAULT TRUE`;
   - `created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP`;
   - `updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`.
2. Изменить запросы чтения:
   - `rooms`: `WHERE is_active = TRUE ORDER BY sort_order ASC, id ASC`;
   - `gallery_items`: `WHERE is_active = TRUE ORDER BY sort_order ASC, id ASC`.
3. Обновить методы создания и редактирования, чтобы они поддерживали новые поля.
4. Для существующих строк заполнить `sort_order` на основе текущего `id`.

Критерии приемки:

- Порядок карточек можно менять без пересоздания записей.
- Неактивные карточки не отображаются на публичной странице.
- У комнат и галереи есть timestamps.

Приоритет: высокий.

### 5.5. Этап 4. Добавить ограничения целостности

Задачи:

1. Добавить `CHECK`-ограничения для `rooms`:
   - `capacity > 0`;
   - `size_sqm > 0`;
   - `room_count > 0`.
2. Добавить ограничения длины и обязательности для ключевых строковых полей с учетом реального UI.
3. Для `gallery_items.tone` ввести допустимые значения через `CHECK` или вынести их в справочник `gallery_tones`.
4. Для `site_content.email` и `booking_url` оставить прикладную валидацию в Go, но определить требования к формату в сервисном слое.

Критерии приемки:

- Невозможно сохранить комнату с нулевой или отрицательной вместимостью.
- Невозможно сохранить неизвестный `tone`, если выбран вариант с `CHECK`.
- Ошибки валидации обрабатываются приложением предсказуемо.

Приоритет: высокий.

### 5.6. Этап 5. Нормализовать медиа

Задачи:

1. Создать таблицу `media_assets`:
   - `id BIGINT PRIMARY KEY AUTO_INCREMENT`;
   - `path VARCHAR(255) NOT NULL`;
   - `alt_text VARCHAR(255) NOT NULL DEFAULT ''`;
   - `mime_type VARCHAR(100) NULL`;
   - `width INT NULL`;
   - `height INT NULL`;
   - `size_bytes BIGINT NULL`;
   - `created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP`;
   - `updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`;
   - уникальный индекс по `path`.
2. В `site_content` добавить `hero_media_id BIGINT NULL`.
3. В `rooms` добавить `media_id BIGINT NULL`.
4. В `gallery_items` добавить `media_id BIGINT NULL`.
5. Добавить внешние ключи на `media_assets(id)`.
6. На переходный период оставить старые `*_image_path` поля и заполнить `media_assets` из существующих путей.
7. После обновления кода удалить старые строковые поля отдельной миграцией.

Критерии приемки:

- Изображения имеют единое описание и alt-тексты.
- Один файл можно переиспользовать в нескольких местах.
- Приложение сохраняет обратную совместимость на время миграции.

Приоритет: средний.

### 5.7. Этап 6. Улучшить хранение `scrape_snapshots`

Задачи:

1. Добавить поля:
   - `content_hash CHAR(64) NULL`;
   - `status VARCHAR(32) NOT NULL DEFAULT 'stored'`;
   - `html_size_bytes BIGINT NULL`;
   - `markdown_size_bytes BIGINT NULL`.
2. Добавить индексы:
   - `idx_scrape_snapshots_created_at(created_at)`;
   - `idx_scrape_snapshots_source_created(source_url, created_at)`;
   - `idx_scrape_snapshots_hash(content_hash)`.
3. Изменить выбор последнего снимка на `ORDER BY created_at DESC, id DESC`.
4. Определить политику хранения:
   - хранить последние N снимков на источник;
   - или хранить снимки за последние N дней;
   - или переносить большие `html`/`markdown` в файловое/S3-совместимое хранилище.
5. Добавить задачу очистки старых снимков.

Критерии приемки:

- Последний снимок выбирается по времени создания, а не только по `id`.
- Можно быстро найти историю по источнику.
- Есть понятная политика роста таблицы.

Приоритет: средний.

### 5.8. Этап 7. Разделить контентные блоки

Задачи:

1. Оценить, нужно ли дробить `site_content` на тематические таблицы:
   - `site_settings`;
   - `hero_blocks`;
   - `contact_details`;
   - `about_sections`;
   - `booking_settings`.
2. Если сайт остается одностраничным, допустимо оставить `site_content`, но добавить `content_key`.
3. Если планируется несколько страниц или несколько объектов размещения, создать таблицу `properties` и связать с ней контент:
   - `properties`;
   - `property_site_content`;
   - `property_rooms`;
   - `property_gallery_items`.

Критерии приемки:

- Для одностраничного сайта схема остается простой.
- Для будущего multi-property сценария есть понятная точка расширения.
- Решение зафиксировано в README или архитектурной заметке.

Приоритет: средний.

### 5.9. Этап 8. Вынести hardcoded-контент из шаблонов

Критерии приемки:

Приоритет: средний.

### 5.10. Этап 9. Добавить аудит изменений

Задачи:

   - `id BIGINT PRIMARY KEY AUTO_INCREMENT`;

   - `action VARCHAR(32) NOT NULL`;
   - `old_value JSON NULL`;
   - `new_value JSON NULL`;

Приоритет: низкий для текущего публичного сайта, высокий для будущей админки.

## 7. Минимальный набор изменений для ближайшего релиза

Если нужно быстро улучшить БД без большой перестройки, достаточно выполнить:

1. Миграции вместо ручного `EnsureSchema`.
2. `site_content.content_key` с уникальным индексом.
3. `rooms.sort_order`, `rooms.is_active`, `rooms.created_at`, `rooms.updated_at`.
4. `gallery_items.sort_order`, `gallery_items.is_active`, `gallery_items.created_at`, `gallery_items.updated_at`.
5. `CHECK`-ограничения для числовых полей `rooms`.
6. Индекс `idx_scrape_snapshots_created_at`.
7. Интеграционный тест на создание схемы и загрузку главной страницы.

Этот минимум устранит главные риски: неявный singleton, зависимость сортировки от `id`, отсутствие миграций и слабый контроль целостности данных.
